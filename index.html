LabelIX V1 — Product Proposal
What we are building, what we are not building, why, and how we got to this scope. Written for internal leadership and external technical review. No marketing language.
⚠
Important context before reading this document
The previous iteration of this product was scoped as an agentic AI platform. After rigorous architectural review, we have reset the scope to a deterministic, regulated workflow application. This is not a step backward. It is the correct foundation. Agentic AI capabilities can be layered on top of a working, validated system. They cannot substitute for one.
The problem we are solving

Medical device manufacturers manage hundreds of label variants across global markets. Every label change — a manufacturer address update, a symbol addition, a regulatory requirement change — must pass through multiple departments for approval before it can reach manufacturing. Today this process runs on email chains, shared drives, and manual review. There is no central system, no audit trail, no visibility, and no enforcement. Changes take weeks. Errors reach manufacturing.

J&J already uses GlobalVision for label comparison. The pain is not in the comparison step — it is in everything around it. Initiating a change. Routing it to the right people. Getting 6 departments to sign off. Tracking where it is blocked. Releasing the right version to manufacturing. LabelIX replaces that entire surrounding process and eventually replaces GlobalVision too.

What LabelIX V1 is
✓
One sentence
LabelIX V1 is a regulated label lifecycle management system — a structured workflow application that handles change initiation, label metadata management, automated comparison and proofreading, multi-department approval, and manufacturing release, built to medtech compliance standards from day one.
What the system owns
Structured label metadata — field values per variant, per market
Change request lifecycle — initiation through manufacturing release
Comparison and proofreading — YOLO-based, deterministic, no LLM dependency
Approval workflow — multi-department routing, e-signature, audit trail
Proof reports — structured, immutable, SHA-256 hashed
What the system does not own
Label artwork — InDesign files stay with the design team
Print-ready file generation — output still comes from InDesign
Live PLM sync — CSV import only in v1
Agentic AI — no autonomous decision-making, no LLM in core workflow
Canvas-based label editing — artwork stays in InDesign
The InDesign constraint — why the label editor works differently than expected
⚠
This is the most important architectural decision in v1
J&J's label source files are InDesign documents exported to PDF. InDesign is a proprietary format — you cannot parse or edit .indd files programmatically. LabelIX cannot own the artwork in v1. Attempting to do so would require replacing J&J's design toolchain before the product has proven its value — a deal-breaker for any regulated customer.

The v1 model works as follows. LabelIX stores structured field metadata for each label variant — lot number format, expiry format, manufacturer address, symbol set, UDI, market-specific fields. When a change is required, the user declares what changed and what the new values are in LabelIX. That structured change request goes to the design team. The design team makes the edit in InDesign and uploads the new exported PDF to LabelIX. LabelIX runs the comparison engine to verify that the detected changes match the declared changes. If they match, the proof is clean and enters the approval workflow.

This is less visually impressive than a canvas editor. It is more buildable, more auditable, and more aligned with how J&J's artwork workflow actually operates today. A canvas editor that owns print-ready file generation is a v3 capability — after the product is trusted, the team is larger, and the label template library has been built.

NLU chat — technology decision
ℹ
This is not an AI chatbot — it is a structured command interface
The NLU chat in v1 handles navigation and read-only data queries. 15 to 20 intents. Rules-based classifier or a lightweight keyword model — no DistilBERT, no training data required, no cold-start problem. Write actions (approve, edit, submit) stay in the UI with proper confirmation flows and audit trails. Chat is the fastest way to navigate and query. It is not a decision-making interface.
What chat does in v1
Navigate — "open SKU 103565000", "show pending approvals"
Query data — "how many labels are pending?", "status of REQ-042"
Start a comparison — "compare labels for REQ-042"
Show a report — "show report for REQ-042"
What chat does not do in v1
No write actions via chat — no approvals, edits, or submissions
No multi-turn context — each message is independent
No regulatory question answering — that is v2 with RAG
No DistilBERT — no training data needed, no cold-start problem

For a controlled pilot with a medical device manufacturer — not full production, but real label workflows with their QA team watching — what is the minimum compliance bar we need to meet before go-live?

Our audit log captures actor, action, entity, before state, after state, and timestamp for every system event in an append-only PostgreSQL table. We would rather design this correctly upfront than rebuild it when J&J's quality team runs their first audit.

We have assumed that J&J's label artwork lives in InDesign files managed either internally or by a design agency, and that they export to PDF for comparison and approval. What is not in V1 — and why
These are not omissions. They are deliberate deferrals with specific reasons. Each one has a path to a future version.
Canvas label editor
Management believes this is the go-to-market angle. The reality: J&J's artwork is InDesign-sourced. We cannot own print-ready file generation in v1 without replacing their design toolchain. A canvas editor that cannot produce a validated print file is a prototype, not a product. This is v3 — after the product is trusted, the team is larger, and the label template library is built.
Live PLM connector
Windchill runs on-premise behind a corporate firewall. IT security approval, custom field mappings, outdated documentation. What looks like 2 weeks of work regularly becomes 3 months. CSV import solves the v1 data problem cleanly. PLM connector is v2 when the customer relationship is established and we have IT access.
Agentic AI / LLM in core workflow
The previous architecture placed Claude as the orchestrator of the entire comparison pipeline. This cost $600/month on a pre-revenue product and introduced a non-deterministic element into a regulated workflow. Claude is now bounded to AMBIGUOUS findings only. The core workflow is deterministic Python. No autonomous decision-making.
DPO learning pipeline
DPO requires a base model to fine-tune and thousands of labelled preference pairs. Neither exists yet. The data collection infrastructure (structured human decisions) is in v1. The actual fine-tuning is v3+ when there is enough data and a base model to fine-tune against. Listing it as a v1 feature was dishonest.
Self-serve onboarding and Stripe billing
Managed onboarding — internal team sets up each customer — is faster to build and sufficient for 3–5 pilot customers. Self-serve adds complexity before we know what the onboarding flow should be. Stripe billing is v2 when customer count justifies the infrastructure.
IQ/OQ/PQ validation documentation
This is not deferred — it runs in parallel from Week 1 with the regulatory consultant. It is not a product feature but it is non-negotiable for any medtech customer using this in production. Every feature decision affects what the OQ test scripts look like. The consultant must be engaged before development starts, not after.
Comparison Engine
Engine state today — and the plan to production accuracy
Honest assessment of where the engine is, what the known gaps are, and what the improvement plan looks like.
⚠
Current state — honest
The comparison engine works well on clean vector PDFs exported from InDesign. It produces noise and false positives on scanned inputs and complex label layouts. The root cause of most false positives is the image-first PDF pipeline — text is being OCR'd from rasterised images instead of extracted from the vector layer. This is a known fix. The Claude orchestrator has been paused due to $600/month API cost. Replacing it with a deterministic Python orchestrator is the first engineering task in v1.
What works today
YOLO symbol detection — 52 ISO classes, 95.6% mAP@50 on training data
Barcode decoding — ZXing + pyzbar, self-validates against printed digits
Image diff — OpenCV + SSIM, handles skewed scans
PDF report generation — structured output, findings by category
Clean PDF comparison — works reliably on InDesign-exported PDFs
Known gaps today
Image-first PDF pipeline — text OCR'd from raster instead of extracted from vector
Scanned input noise — false positives on complex scanned label layouts
YOLO accuracy on J&J's real label portfolio — not yet validated
Claude dependency in orchestrator — paused due to cost, not yet replaced
No held-out accuracy measurement — mAP@50 on training data is not a production accuracy claim
